# Company Overview — Letta Features Company

## Executive summary

This company built a lightweight digital twin platform that turns Letta from a reactive assistant into a proactive operating system. The first pillar is **Schedules**, which holds reminders, follow-ups, and recurring tasks until the right moment and then resurfaces them with context intact. The second pillar is **Skills**, which packages repeatable behaviors into reusable capability modules. The third pillar is **Channels**, which extends the twin into Slack, Telegram, and Discord so the agent can work where teams already communicate.

What makes the build notable is that it is not a single demo artifact; it is a complete delivery stack. The repo includes a file-backed Python API, a static dashboard generator, a hidden/local task-scheduler path, Docker and Compose packaging, GitHub Actions CI/CD, and GitHub Pages publishing for the dashboard. The architecture is intentionally small and inspectable, but it still covers authentication, rate limiting, request logging, dashboard generation, installer wiring, and regression testing. In other words, the company did not just describe the product — it assembled the operational machinery around it.

The work also shows a deliberate company-building process across five rounds of execution. The same core agent pool was reused and re-roled across DevOps, Sales, QA, Architecture, Documentation, and Security work, with 20 explicit role switches logged across the effort. That reuse created continuity: early research informed feature selection, sales collateral fed product framing, DevOps validated hidden execution, and architecture consolidated the final system view. The result is a coherent digital twin company story with three shipped feature pillars, a working delivery pipeline, and a visible hardening backlog for Round 5.

## Technical achievements

- Built a **file-backed schedules platform** with a standard-library HTTP server, a schedule registry, request logging, API authentication, rate limiting, and health checks.
- Built a **static dashboard pipeline** that reads the latest schedule evidence and renders `dashboard/index.html` for local use and GitHub Pages publication.
- Built a **skills manager** that supports skill registration, lifecycle control, disk-based loading, execution logging, and failure handling.
- Built a **channels manager** that normalizes outbound payloads for Slack, Telegram, and Discord and records delivery receipts.
- Established **Docker and Compose packaging** so the API and dashboard can be run locally in a repeatable way.
- Established **GitHub Actions CI/CD** so tests run automatically and the dashboard can be deployed through Pages.
- Added **documentation coverage** across architecture, product specs, roadmap, sales collateral, demo instructions, and final reports.
- Added **negative and kill-path tests** to prove the feature surfaces fail safely when inputs, files, or network conditions break.

## Revenue model

The company monetizes through a **land-and-expand** model:

| Product | Pricing tier | Break-even view | Monetization note |
| --- | --- | --- | --- |
| **Schedules** | Starter: **$29/user/month**; Team: **$99/workspace/month**; Business: custom | At a $60/hour loaded labor rate, Starter breaks even at about **0.5 hours** saved per month; Team breaks even at about **1.7 hours** saved per month | Best positioned as a proactive productivity layer that reduces missed follow-ups and admin time |
| **Channels** | Free: **1 channel**; Paid: **$49/month** for unlimited channels | Breaks even at about **0.8 hours** saved per month at the same labor rate | Monetizes multi-workspace and team comms use cases |
| **Skills** | Foundation now; premium/team-shared/marketplace pricing later | Break-even comes from faster task completion and higher reuse, rather than a single fixed seat price | Designed to become the extension layer for premium capability packs and team sharing |

The Northstar-style Schedules case study shows the model’s upside clearly: a few dozen hours of monthly time savings can produce thousands of dollars in value, which makes the subscription cost easy to justify. The revenue motion is therefore not just subscription-based; it is value-capture based on time saved, fewer misses, and better agent reuse.
